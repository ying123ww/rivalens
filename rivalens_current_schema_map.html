当前已有结构

现在代码已经从泛化事实表切到“先选行业 schema，再结构化竞品知识”的主线。

01 当前已有 EvidenceItem 保存来源、标题、URL、摘要、摘录、采集时间和置信度。
02 当前已有 ActiveKnowledgeSchema 由 SchemaSelectionAgent 选择行业、core fields 和 industry extensions。

现在的关键问题

现在主线已经从 ProductFact 迁走。剩下的问题是：抽取还偏规则化，ReportWriter 还没有输出 ReportDocument。

不再用一个通用 ProductFact 兜底，因为它不能稳定表达功能树、定价模型、用户画像和行业扩展。

推荐目标 Schema

这套结构更符合“知识结构化：定义竞品知识 Schema，Agent 产出必须符合 Schema”的要求。 但它应该是 Core Schema，不是所有行业共用的唯一终点。

对象 它回答的问题 核心字段 由哪个 Agent 产出
EvidenceItem 资料从哪里来，原文片段是什么。 id, source_id, url, title, quote, retrieved_at, source_type, confidence CollectionAgent
CompetitorKnowledge 这个竞品的结构化画像是什么。 competitor_id, feature_tree, pricing_model, user_personas KnowledgeStructuringAgent
FeatureTree 竞品有哪些功能，属于哪个功能域。 category, feature_name, availability, maturity, evidence_ids KnowledgeStructuringAgent
PricingModel 套餐、计费单位、价格可见性和企业版策略是什么。 plans, billing_unit, price, currency, pricing_visibility, included_features KnowledgeStructuringAgent
UserPersona 竞品服务什么用户、痛点和购买触发点是什么。 segment, needs, jobs_to_be_done, buying_triggers, evidence_ids KnowledgeStructuringAgent
AnalysisClaim 结构化知识说明了什么竞争含义。 claim, dimension, competitors, reasoning, evidence_ids, confidence AnalysisAgent
ReportDocument 报告每句话绑定了哪些 claim 和 evidence。 sections, blocks, spans, claim_ids, evidence_ids ReportWriterAgent

行业 Schema 决策机制

不同行业的竞品分析重点不同，所以 Rivalens 应该采用“通用核心 + 行业扩展 + 受约束 AI 补充”的结构。

01 官方分类 Industry Taxonomy 从官方 PDF 或配置文件读取行业分类，作为客观一级边界。
02 请求识别 IndustryClassifier 根据用户问题、竞品名称、产品描述判断候选行业和置信度。
03 Schema 选择 SchemaRegistry 加载 Core Schema 和候选行业扩展，如 SaaS、消费品、医疗、金融等。
04 受约束补充 SchemaRefinementAgent AI 只能在证据和 taxonomy 约束下补充细分维度，并写明依据。
05 质检冻结 SchemaReview 通过后生成本次任务的 ActiveKnowledgeSchema，后续 Agent 都按它输出。

Schema 不是随机生成

AI 可以提出行业细分，但必须输出来源、理由、适用范围和置信度。没有证据支持的字段不能进入 active schema。

{
  "candidate_industries": [
    {
      "industry": "productivity_saas",
      "confidence": 0.84,
      "signals": ["Notion", "workspace", "SAML SSO"]
    }
  ],
  "schema_extensions": [
    {
      "field": "security_compliance",
      "reason": "Enterprise SaaS 采购通常关注 SSO、SCIM、audit log。",
      "evidence_ids": ["ev_1", "ev_4"],
      "confidence": 0.78
    }
  ]
}

最终使用 ActiveKnowledgeSchema

后续 Agent 不直接面对“所有可能字段”，而是面对本次任务冻结后的 schema，保证输出稳定。

{
  "core": ["feature_tree", "pricing_model", "user_personas"],
  "industry": "productivity_saas",
  "extensions": [
    "security_compliance",
    "admin_governance",
    "integration_ecosystem",
    "ai_capability"
  ],
  "version": "task_schema_2026_05_24"
}

SchemaSelectionAgent 与 KnowledgeStructuringAgent 的联系

现在的连接点是 ActiveKnowledgeSchema、ResearchBranch 和 EvidenceItem。CollectionAgent 会按 schema 生成 root branches， BranchReviewAgent 审查是否展开子分支，ResearchEngineEvidenceCollector 会把 ResearchEngine sources 标准化为 EvidenceItem，KnowledgeStructuringAgent 再消费 schema 和 evidence 来产出 CompetitorKnowledge。

目标数据链路

以后报告的每一句话都应该能回查到 claim，再回查到结构化竞品知识和原始证据。

资料层 EvidenceItem 网页、数据库、PDF 或评论来源里的可引用证据片段。
知识层 CompetitorKnowledge 功能树、定价模型、用户画像，字段稳定且能横向比较。
分析层 AnalysisClaim 基于知识结构得出的竞争判断，而不是凭空写观点。
表达层 ReportDocument section/block/span 结构，每个 span 绑定 claim 和 evidence。
反馈层 FeedbackEvent 用户可以针对句子、claim 或 evidence 反馈，进入修订闭环。

参考项目的共同做法

我们参考的是具有相似职能的 agent / routing / schema-guided extraction 项目：它们通常不让 AI 随机生成规则， 而是把 AI 放进“候选集合、路由、schema registry、反思和质检”的流程里。

参考模式 他们通常怎么做 Rivalens 应该怎么落地
Handoff / Specialist Agents 先显式注册可交接的专家 Agent，模型只能在候选 handoff 中选择，handoff 输入是结构化参数。 显式注册行业 schema 和行业专家，不让模型无限发散；低置信度时返回 unknown 或 request_more_evidence。
Context-aware Routing Router 只做分类和路由，把 category 写入上下文，后续图节点根据 category 执行。 新增 IndustryClassifierAgent，只输出候选行业、置信度、signals，不同时做竞品分析。
Semantic Route Matching 每个 route 有 aliases 和 example utterances，用语义匹配筛出候选类别，无法匹配则返回 None。 官方 PDF 解析成行业 taxonomy 后，为每个行业生成 aliases、example_queries、known_competitors。
Schema Repository 预定义 schema repository，允许自定义 schema，但新增 schema 必须是可验证结构。 建立 schema_registry：core schema 固定，industry extensions 版本化，AI 只能提出 candidate_extensions。
Domain Hint 如果用户或上下文已给出领域，就把 domain_hint 传给研究或抽取流程。 用户明确说行业时直接作为强信号；未明确时先用 query、竞品、初步证据推断。
Reflection Loop 先用默认 schema 抽取，再检查缺失字段，生成补充搜索 query 或 refinement。 先跑 Core Schema，再让 SchemaReviewAgent 检查行业字段缺口，必要时回到 CollectionAgent 补采。

不推荐

让 AI 直接根据用户请求自由生成一整套行业 schema。这样结果看似聪明，但难复现、难质检、难比较。

{
  "industry": "AI 自己猜的行业",
  "schema": {
    "random_field_1": "...",
    "random_field_2": "..."
  }
}

推荐

AI 只能在官方 taxonomy 和 schema registry 中选择、解释、补充候选项，最后冻结为 ActiveKnowledgeSchema。

{
  "selected_industry": "productivity_saas",
  "source": "official_taxonomy + semantic_route + evidence",
  "registry_extensions": ["security_compliance"],
  "candidate_extensions": [
    {
      "field": "workspace_migration",
      "reason": "多条证据显示迁移能力影响企业采用。",
      "evidence_ids": ["ev_7", "ev_12"],
      "confidence": 0.74,
      "approved": false
    }
  ]
}

Agent 应该如何构建

Agent 不应该按底层 report type 命名，而应该按 schema 生产责任命名。每个 Agent 都有固定输入输出。

PlanningAgent

输入用户任务，输出竞品范围、候选行业、分析维度、需要填充的 schema 区域。

SchemaSelectionAgent

读取官方行业 taxonomy，选择 Core Schema 与行业扩展，冻结本次任务的 ActiveKnowledgeSchema。

CollectionAgent

维护 research branch frontier，调用 BranchReviewAgent 和 ResearchEngineEvidenceCollector，只产出 EvidenceItem[]，不负责分析。

KnowledgeStructuringAgent

把 evidence 填入 CompetitorKnowledge，不再经过 ProductFact 兜底。

AnalysisAgent

基于 CompetitorKnowledge[] 做横向比较，输出 AnalysisClaim[]。

QualityAgent

审查 schema 字段、claim、evidence 绑定和置信度，输出 QualityFinding[]。

ReportWriterAgent

输出 ReportDocument，Markdown/HTML/PDF 都只是渲染物。

下一步改造顺序

这部分是我建议你接下来真正动代码的顺序，先把业务 schema 稳住，再改 Agent。

第一优先级：新增 Core Schema 与 SchemaRegistry

先定义所有行业都共用的 CompetitorKnowledge 核心字段，再为行业扩展预留 registry。 不要把 SaaS、消费品、医疗、金融等不同业务重点写死到同一个大对象里。

第二优先级：新增行业分类与选择 Agent

从官方 PDF 或配置文件加载 IndustryTaxonomy，让 SchemaSelectionAgent 输出候选行业、 置信度、依据和本次任务的 ActiveKnowledgeSchema。

第三优先级：加入候选字段审查机制

AI 可以提出 candidate_extensions，但必须带 reason、signals、evidence_ids、confidence。 SchemaReviewAgent 通过后才能进入本次 active schema。

第四优先级：精细化 CompetitorKnowledge 抽取

当前已经定义 FeatureTree、PricingModel、UserPersona。下一步要让抽取结果更稳定， 并按 active schema 中的行业扩展填充字段。

第五优先级：完善 KnowledgeStructuringAgent

让它从 EvidenceItem 中分别抽取 feature、pricing、persona、industry extension， 并把每个字段绑定 evidence_ids。

第六优先级：让 AnalysisAgent 消费知识 Schema

AnalysisAgent 已不再围绕 ProductFact 做简单 claim；后续要增强横向对比、机会点和风险判断。

第七优先级：新增 ReportDocument

Writer 产出结构化文档，Publisher 负责渲染。这样之后才好做 hover 高亮、证据跳转和用户反馈。

当前结论：消息结构化已经完成第一步；下一步应从 Core Schema + Industry Schema + ActiveKnowledgeSchema 三层升级业务知识结构。
