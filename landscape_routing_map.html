Rivalens control plane

Landscape 搜索结果如何变成 routing decision

Landscape 和 focused 现在不是靠 action 区分，而是靠 search_stage 和 stage_contract 区分执行面。ResearchRoutingAction 是共享路由词汇； landscape 只写 source-universe observation，focused / verification 才写入 evidence plane。

真源规则

  • Full state landscape_assessments 保存完整 observation 和 routing。
  • Stage stage_contract.produces_evidence 明确是否能产证据。
  • Artifact 只保存 observation / routing / replay_ref 的轻量投影。
  • Evidence landscape candidate 不进入 EvidenceItem，除非后续 focused 深采。

End-to-end control flow

先看 stage contract，再看六类 routing

action 决定下一步路由意图；stage contract 决定这次运行的输出落点。真正可用于分析的证据，必须经过 focused 或 verification collection 变成 accepted EvidenceItem。

Returned structure

Landscape 搜索回来的结构

Full state 保存完整对象；artifact 只保存可回放投影。两者都指向同一个 landscape_assessment_id。

Full state: LandscapeAssessment

router 只从这个对象派生 decision。它保存 observation、候选 follow-up、scorecard 和仲裁结果。

Six ResearchRoutingDecision classes

六类 routing 维度

这些是 landscape、focused 和 verification 共用的 control taxonomy，不等于六个搜索接口。router 会给每类候选打分， 最高分写入 decision；stage_contract 决定当前 observation 来源和能不能产出 evidence。

scope_refinement

范围重整

当搜索入口不清楚，或维度过宽，需要先改 query 或拆子维度。

  • query_refinement: 没有 candidate source。
  • dimension_decomposition: 例如 competitive_moat 拆成 switching cost、ecosystem lock-in 等。
entity_resolution

实体消歧

候选 URL 存在，但域名和 competitor 绑定不安全，可能是同名公司、目录页或错误产品。

  • competitor_disambiguation: 生成 focused official-site task。
source_discovery

源类型补采

expected source type 没覆盖，尤其是 pricing、official、docs 这类关键来源。

  • source_type_search: landscape 阶段发现 source type 缺口。
  • coverage_gap_search: focused 后 CoverageReviewer 发现覆盖缺口。
evidence_extraction

定向抽取

已经找到具体 URL，后续任务不再泛搜，而是把 URL 下推成 source_urls。

  • targeted_url_extract: focused child task 走 URL 抽取。
claim_verification

声明验证

这类不是 LandscapeReviewer 常规产生，而是 ClaimSupportReviewer 在 weak/unverifiable claim 后进入 verification queue。

  • evidence_check: 搜索支持或反驳具体 claim 的证据。
stop

停止扩展

没有可执行 follow-up，或预算/深度限制阻止继续扩展。

  • budget_stop: depth 或 branch budget 阻断。
  • sufficient_stop: source universe 足够清楚且无后续任务。
  • no_viable_followup: 无有效后续任务。

After routing

六类分支之后怎么走

下面按当前代码行为画清楚每类 decision 对 ResearchTask、EvidenceItem、review loop 的影响。

Decision 进入条件信号 生成的 follow-up 后续路径 是否进 EvidenceItem
scope_refinement 无 candidate source，或维度过宽需要拆解。 query_refinement 生成 landscape child；dimension_decomposition 生成 focused child dimensions；统一写入 follow_up_task_specs。 source_collection -> landscape/focused child -> 再次 assessment 或 evidence review 否。只有 focused child 深采后才会生成证据。
entity_resolution candidate domain 无法绑定 competitor，存在同名或错站风险。 competitor_disambiguation focused task，目标 source type 通常是 official_site。 focused official-site search -> EvidenceItem -> EvidenceQualityReviewer 后续 focused 结果可以进。
source_discovery missing_source_types 非空，关键 source type 缺口会提高分数。 source_type_search 或 coverage_gap_search focused task。 focused search -> EvidenceItem -> CoverageReviewer -> 可能继续补采 后续 focused 结果可以进。
evidence_extraction 存在高置信 candidate URL，且缺口不压过 URL 抽取。 targeted_url_extract，带 target_urls。 focused URL extract -> ResearchEngine.source_urls -> EvidenceItem 后续定向抽取结果可以进。
claim_verification analysis claim 被评为 weak、contradicted 或 unverifiable。 verification_task_queue，search_stage = verification。 claim_support_review -> source_collection verification pass -> EvidenceItem -> claim support 再评估 verification 采到的证据可以进。
stop 预算阻断、已经足够、或没有可执行 follow-up。 无 child task；写入 assessment 和 artifact routing。 branch.status = stopped -> 等待其他 branch 或进入后续 agent 否。
