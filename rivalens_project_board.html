当前代码主线：landscape / focused / verification research control

Rivalens 现在是“先发现入口，再采证据，最后按 claim 验证”的研究链路

当前主流程已经调整为： PlanningAgent 生成竞品和 10 个 analysis dimensions，CollectionAgent 按 competitor x dimension 建 research branches，并按 search_stage 执行 landscape、focused 或 verification。landscape 只发现入口， focused 才产出 EvidenceItem，ClaimSupportReviewer 会把弱 claim 推回一次 verification 补采。

当前状态

  • 已接上DAG 顺序是 scope_planner -> source_collection -> knowledge_structuring -> dimension_analysis -> claim_support_review。
  • 已接上landscape 产出 LandscapeAssessment 和 focused_task_specs，不会伪装成最终证据。
  • 已接上focused 和 verification 才生成 EvidenceItem，再过 evidence / coverage review。
  • 已接上ClaimSupportReviewer 会为 weak / unverifiable claim 创建一次 verification_task_queue。
  • 已接上dimension_split_suggestions 会生成带血缘的 split child dimension tasks。
  • 已接上competitor_disambiguation 会生成专用 follow-up task，不再落回通用 refinement。
  • 漏洞claim support 现在是词面 overlap 检查，还不是严格 entailment / contradiction 判断。
  • 下一步补更强 claim support、verification span 判断和更强 claim 生成。

Research 逻辑架构

当前不是固定三轮，而是每个 ResearchTask.search_stage 决定这次采集的目标： landscape 发现 source universe，focused 采 EvidenceItem，verification 只验证具体 claim。

当前 DAG 流转

这是 rivalens/workflows/competitive_analysis.py 的主链路。关键变化是 analysis 现在在 knowledge 之后， claim support 在 writer 之前，并且只有 weak / unverifiable claim 能触发一次 verification 回采。

三阶段衔接图

现在 landscape、focused、verification 都是 ResearchTask.search_stage。它们共享 branch lineage， 但输出对象不同，所以不能把 landscape 结果直接送进 analysis。

  • Root Branch Competitor x Analysis Dimension

    PlanningAgent 给出竞品和确认维度，CollectionAgent 生成 root branch、ResearchBrief 和 ResearchTask。

    • landscape source-universe discovery

      用于 market_growth、competitive_moat 或 high-risk / source universe 不清楚的维度。输出候选 source、source type、消歧状态和 follow-up specs。

      LandscapeAssessment: candidate_sources, focused_task_specs, next_action
      • needs_focused_collection candidate source -> focused child branch

        前两个候选 URL 或缺失 source type 会变成 focused task，带 parent_task_id 保留血缘。

      • 已接上 needs_dimension_split / disambiguation 会生成 child task

        split suggestions 会变成新的 child dimensions；消歧会生成 competitor_disambiguation follow-up task。

    • focused evidence collection

      默认阶段。大多数 pricing、docs、review、marketplace 维度直接从 focused 开始，或者由 landscape follow-up 生成。

      EvidenceItem -> EvidenceQualityReviewer -> CoverageReviewer
      • accepted evidence 进入 KnowledgeStructuringAgent

        EvidenceItem 保留 branch_id、collection_task_id、research_task_id、dimension_id 和 URL，用于后续 claim 回查。

      • coverage gap collect_more / refine_query

        CoverageReviewer 可以生成 follow_up_task_specs，继续扩一个 depth 以内的 child branch。

    • verification claim-driven follow-up

      不是普通覆盖率补采。只有 ClaimSupportReviewer 发现 weak / unverifiable claim 时才会把 verification_task_queue 推回 CollectionAgent。

      ClaimSupportReview -> verification_task_queue -> verify_* branch
      • bounded loop 最多一轮 verification

        route_after_claim_support 只有在 queue 非空且 verification_rounds 为 0 时回 source_collection，避免无限循环。

读图方式

1. landscape 不是证据

它只产生 source universe、query refinements 和 focused_task_specs。只有 focused / verification 的结果会变成 EvidenceItem。

2. focused 是主采集

它接受 root branch、landscape follow-up 或 coverage follow-up，产出可被 knowledge / analysis 使用的 accepted evidence。

3. verification 只服务 claim

它从 ClaimSupportReview 生成，目标是支持或推翻具体 claim，不负责扩大普通维度覆盖。

4. 当前漏洞

strong entailment、verification span 判断、report-level QA 和更强 analysis claim 还需要继续补。

当前核心对象

现在最重要的不是前后端，而是这些对象之间的血缘：task、branch、evidence、claim、claim support review 必须能互相回查。

ResearchBrief / ResearchTask

ResearchBrief 定义 competitor x dimension 的目标；ResearchTask 带 search_stage、query、parent_task_id 和 generated_from_gap。

LandscapeAssessment

只描述 source universe：candidate_sources、discovered_source_types、competitor_disambiguation、dimension_split_suggestions 和 follow-up specs。

EvidenceItem / CoverageAssessment

focused / verification 产出 EvidenceItem；EvidenceQualityReviewer 和 CoverageReviewer 决定 accepted evidence 与补采任务。

CompetitorKnowledge

KnowledgeStructuringAgent 只消费 accepted evidence，结构化每个竞品的公开证据和维度知识。

AnalysisClaim

AnalysisAgent 基于 CompetitorKnowledge 生成 claim，每条 claim 必须绑定 evidence_ids、dimension、competitors 和 confidence。

ClaimSupportReview

ClaimSupportReviewer 标记 supported / weak / contradicted / unverifiable，并为弱 claim 生成一次 verification task。
