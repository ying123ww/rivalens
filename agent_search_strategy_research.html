Agent Search 调研

模型内置搜索、Tavily 等专业 Search API、以及竞品分析场景中的内部/专有数据 API 对比。重点回答：什么时候靠模型自带搜索，什么时候必须接搜索 API，什么时候需要 Multi-agent Search。

调研日期：2026-05-24 范围：OpenAI / Gemini / Groq / Tavily / SerpAPI / Exa / Brave / 竞品分析与 GTM 数据源 结论：竞品分析建议采用 Multi-agent Search + 内部数据治理层
模型内置搜索

OpenAI、Gemini、Groq 都提供或封装了搜索能力，适合快速补充近期公开事实、生成带引用的回答，以及做轻量研究。

专业 Search API

Tavily、SerpAPI、Exa、Brave Search API 更适合可控检索、结构化结果、批量查询、固定成本评估和 agent 工作流编排。

竞品分析数据源

高价值竞品分析不仅依赖公开网页，还需要 CRM、销售通话、win/loss、买家意图、SEO/流量和内部协作数据。

Multi-agent Search

竞品分析建议拆成 PublicWebAgent、SearchApiAgent、MarketDataAgent、InternalIntelAgent 和 Verifier，分别处理召回、权限、成本和核验。

总判断

如果只是回答公开事实、找近期新闻、补充引用，模型内置搜索可以作为第一层能力。如果要做稳定、可观测、可评估、可批量运行的 Agent 搜索，应该接 Tavily、SerpAPI、Exa、Brave Search API 这类专业 Search API，并把它们作为明确的工具层。

如果 Agent 方向是竞品分析，单一搜索源不够。公开网页只能覆盖官网、新闻、发布说明、价格页和评论的表层信息；真正决定分析质量的通常是 CRM、Gong/Chorus 销售通话、Slack/Teams 一线反馈、G2 买家意图、Similarweb/Semrush 流量与关键词、ZoomInfo 公司/联系人图谱等内部或付费专有数据。

1. 官方项目/文档的具体做法

OpenAI Responses API / Agents SDK 内置搜索 Responses API 可使用 web_search 工具；Agents SDK 提供 WebSearchTool，支持用户位置、上下文大小等配置。 适合做公开证据检索和引用生成。若要稳定接竞品数据库、CRM 或自定义搜索源，应把这些系统封装成 function tool 或 MCP tool。 OpenAI Web search；Agents SDK Tools
Gemini API Google Search Grounding 内置搜索 Gemini API 可配置 google_search tool，模型会生成搜索查询、处理结果，并返回 grounding metadata。 适合公开网页事实、近期事件和可引用回答。若要接内部数据，仍需要额外工具或独立 agent。 Gemini Google Search Grounding
Google ADK Gemini Interactions API 工具组合限制 ADK 文档说明 Gemini Interactions API 在某些模式下不支持同一 agent 直接混用内置 Google Search 与自定义函数工具，可通过 wrapper 或 multi-tool 设计处理。 当竞品分析同时需要内置搜索和 CRM/Gong/G2/Similarweb 等工具时，更自然的形态是 Search Router 或多 agent 编排。 ADK Gemini models
Groq Compound 内置搜索 Groq 的 groq/compound 与 groq/compound-mini 支持服务器侧 web search，返回引用和 executed tool 的搜索结果；Groq 文档说明其 web search 底层由 Tavily 提供。 这是“模型 API 自带搜索，但底层仍依赖专业搜索基础设施”的代表。适合低延迟公开网页检索。 Groq Web Search
Groq + Agno 官方集成 外部搜索工具 Groq 的 Agno 示例把 Groq LLM 与 DuckDuckGoTools 组合成 web search agent。 说明即使模型平台有内置能力，官方 agent 示例也常把搜索作为独立工具来调用，便于替换和编排。 Groq + Agno
Tavily API 专业 Search API Tavily Search 通过 POST /search 执行搜索，面向 AI 应用提供搜索结果、答案和可配置参数；文档还提供 research 等 agent 友好的能力入口。 适合作为 PublicWebAgent 的主力 API：比内置搜索更可控，便于记录请求、复现结果、做批量监控和成本统计。 Tavily Search API；Tavily Docs
SerpAPI 专业 SERP API SerpAPI 提供 Google Search Engine Results API，并有官方 web-research-agent 示例：模型发出 search_web function call，再调用 SerpAPI 获取结构化 SERP。 适合需要接近搜索引擎结果页结构、地域/设备参数、SERP 特性和可重复监控的场景。 SerpAPI Search API；web-research-agent
Exa API AI-native Search API Exa 官方定位是“search engine made for AIs”，提供 /search、/contents、/findsimilar、/answer、/research 等能力。 适合语义搜索、相似页面发现、内容抽取和深度研究。对竞品分析，可用于查找相似产品、替代方案、长尾资料和引用材料。 Exa Docs；Exa Search
Brave Search API 独立 Web Index API Brave Search API 面向 agents、chatbots 和 AI-search engines，提供 URLs、文本、新闻、图片等结果，以及面向 LLM 的上下文能力。 适合希望减少对单一大搜索引擎依赖、需要独立索引、需要较高 QPS 或需要定制 reranking/filtering 的检索层。 Brave Search API
Domo Competitive Intelligence AI Agent 竞品分析 Agent Domo 的模板描述了把结构化销售数据与外部竞争情报结合，分析 reviews、product descriptions、news、pricing 等非结构化资料。 官方竞品分析 Agent 明确需要“内部销售数据 + 外部公开/半公开材料”，不是单纯 web search。 Domo CI AI Agent
SimplAI Competitive Analysis Agent 竞品分析 Agent 官方页面称其监控网站变化、广告、招聘、新闻、社媒和财报电话会，并与 Salesforce、HubSpot、Slack、Crayon、Kompyte、Google/Meta Ad Libraries 集成。 这是典型多信号采集：公开网页、广告库、招聘、财报、CRM 和协作工具进入同一竞争情报流。 SimplAI Competitive Analysis Agent
Poggio Revenue Superagent 多源 Agent Poggio 官方页面提到结合 Salesforce、Gong 和大量外部来源，并展示 DBs、APIs、Calls、Web、MCPs 等输入。 面向销售/营收的 Agent 会把 CRM、通话、外部网页和 API 一起用于账户计划，与竞品分析的账户级 brief 很接近。 Poggio

2. 内置搜索 vs 专业 Search API

内置搜索优势

  • 集成最快：模型 API 直接决定是否搜索，应用层只需配置工具。
  • 回答自然：模型可以直接整合搜索结果、生成摘要和引用。
  • 适合轻量研究：公开事实、近期新闻、政策页面、官方公告、简单问答。
  • 成本和架构简单：不需要独立维护搜索服务、缓存层和结果解析链路。

内置搜索限制

  • 可控性弱：查询改写、召回、排序和来源选择较黑盒。
  • 复现难：同一问题在不同时间或不同模型版本下结果可能变化。
  • 批量监控弱：对定时抓取、固定 query set、差异检测和审计不够直接。
  • 不能访问私有数据：CRM、销售通话、内部文档、付费市场数据仍要接工具或 API。

专业 Search API 优势

  • 检索可控：可记录 query、参数、返回结果、时间戳和供应商。
  • 适合 Agent 编排：可并发、可重试、可缓存、可评分、可分源限流。
  • 结果结构化：SERP、网页内容、新闻、图片、相似页面、answer/research endpoints 更容易进入下游处理。
  • 评估友好：可以固定测试集，比较 Tavily、SerpAPI、Exa、Brave 等源的召回、准确率、延迟和成本。

专业 Search API 成本

  • 接入更重：认证、配额、错误处理、缓存、去重和内容抓取都要设计。
  • 供应商差异大：有的偏 SERP，有的偏语义搜索，有的偏 LLM-ready 内容。
  • 仍不等于内部情报：它们通常只能抓公开网页或公开索引，不能替代 CRM/Gong/G2/Similarweb 等数据源。
  • 需要治理：竞品监控容易抓到过时、重复、广告化或二手转述内容，必须有核验层。

3. 竞品分析是否需要 Multi-agent Search

结论

需要。竞品分析的难点不是“能不能搜索网页”，而是如何把不同可信度、权限等级和数据形态的信息合并：公开网页负责事实证据，专业 Search API 负责稳定召回，市场数据 API 负责行业与渠道信号，内部系统负责真实客户反馈，最后由核验层处理冲突、时间衰减和保密边界。

数据/API 类型 典型系统 信息密度 适合回答的问题 接入建议
模型内置搜索 OpenAI web search、Gemini Google Search Grounding、Groq Compound 低到中：覆盖广，适合开放网页证据。 竞品最近发布了什么？官网怎么描述功能？公开价格或条款是否变化？ 作为第一层快速检索和引用层，不作为唯一事实源。
专业 Search API Tavily、SerpAPI、Exa、Brave Search API 中：可控、可批量、可评估，适合 agent 工作流。 每周监控竞品页面变化；批量抓竞品新闻；找相似产品；做多源交叉验证。 建议至少接一个主搜索 API，再保留一个备用源用于召回补充和供应商 fallback。
专有市场/SEO 数据 Similarweb、Semrush、Ahrefs、G2、ZoomInfo、Apollo 中到高：付费专有数据、买家意图、流量、关键词、公司图谱和联系人信号。 竞品流量是否上升？关键词在抢什么？哪些账户在比较我和竞品？哪些公司显示购买意图？ 作为 MarketDataAgent 的输入，和公开网页证据分开计分。
企业一方/内部系统 Salesforce、HubSpot、Gong、Chorus、Slack、Teams、Zendesk、Intercom、Jira、Confluence 最高：真实客户反馈、输赢原因、销售异议、支持问题和产品缺口。 客户为什么输给竞品？销售最常遇到的竞品异议是什么？哪个功能缺口影响成交？ 生产版必须只读优先、权限继承、PII 脱敏、审计日志和来源分级。

4. 推荐架构

公开证据

PublicWebAgent

调用模型内置搜索，快速找官网、新闻、发布说明、价格页、评论和公开文档，并生成带引用的初步事实。

可控召回

SearchApiAgent

调用 Tavily、SerpAPI、Exa 或 Brave，负责批量查询、相似页面、内容抽取、固定 query set 和差异监控。

市场信号

MarketDataAgent

调用 Similarweb、Semrush、G2、ZoomInfo 等专有 API，处理流量、关键词、买家意图、公司图谱和账户信号。

内部情报

InternalIntelAgent

只读接入 CRM、销售通话、Slack/Teams、支持工单和内部知识库，抽取 win/loss、异议、竞品提及和销售反馈。

Verifier / Synthesizer

所有 agent 的结果进入统一核验层：按来源标注 public、search-api、paid-proprietary、internal、confidential；做冲突检测、来源时间衰减、引用合并、PII/保密信息脱敏，然后输出 battlecard、SWOT、账户级竞争态势、周报或销售问答。

5. 落地建议

阶段 建议搜索栈 原因 验收标准
MVP 模型内置搜索 + Tavily 或 Exa + CRM 只读搜索 先覆盖公开证据、可控召回和账户上下文，避免一开始接入所有系统。 能生成带来源的竞品 brief；能记录 search API 请求；能按账户补充 CRM 背景；不会泄露内部敏感字段。
增强版 内置搜索 + 两个专业 Search API + G2 或 Similarweb/Semrush 提升召回和 fallback 能力，并加入竞品分析更需要的买家意图、流量、关键词或市场份额信号。 同一竞品可给出公开证据、市场数据和冲突说明；支持定时监控、去重、来源评分和变化摘要。
生产版 Multi-agent Search + CRM/Gong/Slack + Search API + 专有市场数据 + governance layer 竞品分析的关键差异来自内部 field intel 与付费专有数据，多 agent 可隔离权限、成本、召回和验证逻辑。 具备来源分级、权限继承、PII/保密信息脱敏、冲突检测、引用、审计日志和人工审批点；能输出 battlecard、win/loss 摘要、账户级竞争态势。
