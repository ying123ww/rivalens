面向甲方的端到端技术说明：系统如何从用户需求出发，完成竞品范围确认、公开证据采集、证据审查、知识结构化、分析结论生成、质量复核、报告发布与前端展示。

版本日期：2026-05-31 代码库：/home/ying/projects/rivalens 核心目标：可追溯多 Agent 竞品分析

1. 执行摘要

Rivalens 是一个可追溯的多 Agent 竞品分析系统。它的产品目标不是简单生成一篇“看起来像调研”的文本，而是建立一条可以被复核的证据链：公开网页或本地文件形成 EvidenceItem，证据经过质量审查后进入结构化竞品知识，再生成带 evidence_ids 的分析结论，最后输出带信息索引附录的报告。

当前系统已经具备清晰的 LangGraph DAG 骨架、结构化消息协议、证据质量门、覆盖审查、claim 支持度审查、一次验证回流、报告导出和数据库持久化模型。需要如实说明的是，部分审查逻辑目前仍是规则式启发判断，还不是强语义验证；前端与后端对 Rivalens 专用流程的集成也仍有缺口。

对甲方最重要的价值：Rivalens 可以把“报告结论”回连到“证据 ID、来源 URL、采集任务、分析维度、审查记录”，这使竞品分析具备可解释、可复核、可迭代的工程基础。
项目定位 总体架构 主流程 Agent 职责 数据结构 证据追溯 后端与前端 可观测性 交付与演示

2. 项目定位与业务目标

Rivalens 面向企业产品、战略、市场和投研团队，帮助用户在较短时间内形成结构化竞品分析。系统强调“可信调研链路”，因此核心能力分为四层：

层级面向甲方的解释工程实现载体
范围层明确要比较哪些竞品、行业、分析维度，避免跑题。PlanningAgent、行业方向模板、ActiveKnowledgeSchema
证据层只使用可定位来源的公开证据，保留 URL、摘要、来源类型和采集时间。ResearchEngineEvidenceCollector、EvidenceItem
分析层把证据沉淀为功能、定价、用户画像、行业扩展字段，再形成结论。KnowledgeStructuringAgent、AnalysisAgent
质检层检查证据质量、覆盖缺口和 claim 是否被证据支持，必要时触发补采。EvidenceQualityReviewer、CoverageReviewer、ClaimSupportReviewer
交付层生成可下载报告，并在附录里列出信息索引，便于甲方复核来源。ReportWriterAgent、PublisherAgent、report_export.py

3. 总体架构

系统由 Python 后端、多 Agent 工作流、研究检索引擎、结构化 schema、持久化层和前端界面组成。核心工程边界如下：

工作流编排

主入口是 rivalens/workflows/competitive_analysis.py。它使用 LangGraph StateGraph 定义节点和条件边，保证多 Agent 流程不是散乱调用，而是可解释 DAG。

结构化协议

rivalens/schema/competitive.py 定义 EvidenceItem、CompetitorKnowledge、AnalysisClaim、AgentMessage 等核心协议。

研究引擎适配

ResearchEngineEvidenceCollector 封装底层检索和网页抓取，把外部结果标准化为证据项，避免其他 Agent 直接调用搜索。

报告与导出

ReportWriterAgent 组合 claims、knowledge 和 evidence，PublisherAgent 通过公共导出模块生成 Markdown、HTML、PDF、DOCX。

4. 端到端主流程

当前 LangGraph 主流程如下。注意其中 claim_support_review 存在条件边：如果发现弱支持 claim，且还没有验证过，会回到 source_collection 做一次 claim-driven verification。

1. 用户输入用户提交任务、竞品、文件、行业方向。
2. scope_planner识别行业和竞品，生成或冻结分析维度。
3. source_collection按竞品 x 维度并行采集公开证据。
↓
4. evidence_review检查 URL、竞品匹配、维度匹配、最低来源数、官方来源。
5. coverage_review判断指导问题是否覆盖，生成补采任务。
6. knowledge_structuring把 accepted evidence 组织为竞品知识。
↓
7. dimension_analysis生成带 evidence_ids 的分析 claim。
8. claim_support_review检查 claim 是否被证据支持，必要时进入验证回流。
9. report_writer撰写四章报告并自动追加信息索引。
↓
10. publisher导出 Markdown、HTML、PDF、DOCX 文件。
11. persistence后端尝试落库保存 run、evidence、claims、reviews、events。
12. frontend展示报告、下载路径、运行日志和聊天交互。

5. Agent 职责边界

Rivalens 的 Agent 设计强调职责分离：规划 Agent 不采集网页，采集 Agent 不撰写最终观点，分析 Agent 不直接搜索，写作 Agent 不绕过证据链。这样做的好处是每个环节都可以被记录、测试和替换。

Agent / Reviewer职责不应承担的职责甲方可检查的产物
PlanningAgent确认分析范围、行业、竞品和维度。不直接进行网页搜索，不生成最终结论。行业方向计划、active schema、schema_selection 消息。
CollectionAgent根据维度和竞品采集 evidence，控制深度和扩展预算。不写报告，不直接判断商业结论。research branches、research tasks、evidence items。
EvidenceQualityReviewer判断证据源是否可用，识别缺 URL、竞品错配、维度错配、来源不足等。不负责完整分析覆盖度。accepted / rejected evidence IDs、findings、score。
CoverageReviewer判断该维度问题是否覆盖，生成补采任务。不负责 claim 级别语义支持判断。missing questions、selected follow-up specs、routing decision。
KnowledgeStructuringAgent把证据结构化为竞品知识。不新增外部事实。feature_tree、pricing_model、user_personas、industry_extensions。
AnalysisAgent生成带 evidence_ids 的 claim。不直接调用搜索引擎。AnalysisClaim 列表。
ClaimSupportReviewer检查 claim 是否被证据支持，并触发一次验证回流。不负责终稿排版。ClaimSupportReview、verification queue。
ReportWriterAgent把支持 claim 和证据写成甲方可读报告。不采集新证据，不使用 rejected evidence 支撑重要结论。Markdown 报告和信息索引附录。

6. 核心数据结构

系统用 rivalens/schema/competitive.py 作为唯一核心协议文件。它不是普通类型声明，而是整条可信链路的合同。

结构作用关键字段
EvidenceItem表示一条来源明确的公开证据。id、competitor、branch_id、dimension_id、url、source_type、excerpt
EvidenceReviewResult记录证据质量审查结果。accepted、score、findings、accepted_evidence_ids、required_action
CoverageAssessment记录覆盖度审查和补采决策。missing_questions、selected_follow_up_specs、decision、confidence
CompetitorKnowledge按竞品沉淀结构化知识。feature_tree、pricing_model、user_personas、industry_extensions、evidence_ids
AnalysisClaim最终分析结论的结构化载体。claim、competitors、evidence_ids、branch_id、evidence_review_id
AgentMessageAgent 间结构化交接。sender、receiver、type、payload、artifact_ids、evidence_ids

7. 证据追溯链路

甲方最关心的是：报告里的重要判断能不能回到原始材料。Rivalens 的追溯链路如下：

URL / source网页、文档、检索结果。
EvidenceItem保留 URL、标题、摘要、来源类型、维度、竞品。
EvidenceReviewResult判断接收或拒绝，记录问题。
CompetitorKnowledge把证据变成结构化字段。
AnalysisClaim结论保存 evidence_ids。
Report Appendix信息索引表列出 ID、URL、摘要。

这一链路的关键约束是：下游 Agent 不应绕过 EvidenceItem；重要 claim 必须保留 evidence_ids；报告附录必须让甲方看到每条证据的来源 URL。

8. 后端、前端与持久化

8.1 后端入口

后端通过 FastAPI 提供 HTTP 和 WebSocket。常规报告走 /report/，Rivalens 工作流由 backend/server/rivalens_runner.py 调用 run_competitive_analysis_task，执行后尝试持久化完整状态。

接口 / 模块职责当前状态
/report/生成报告并导出文件。支持 report_type=rivalens 的后端分支，但当前静态前端下拉框没有展示该选项。
/wsWebSocket 运行任务、日志流、聊天。可以启动任务，但一次连接同时只处理一个长任务。
/api/industry-directions预览行业方向计划。会使用 Pydantic payload 校验返回结构。
/api/rivalens快捷调用 Rivalens。目前是硬编码演示问题，不适合正式产品调用。

8.2 持久化

backend/server/persistence.py 定义了从 run、竞品、方向、branch、task、evidence、review、knowledge、claim 到 agent events 的关系表。自动建表默认关闭，需设置 RIVALENS_AUTO_CREATE_TABLES=true 或执行 SQL 脚本。

8.3 前端体验

当前仓库同时存在静态前端和 Next.js 前端。静态前端以通用研究助手体验为主，包含状态面板、历史记录、MCP 配置、报告展示和下载。Rivalens 专用的“证据源查看、Agent 决策回放、人工修正”尚未形成完整前端闭环。

9. 可观测性与质量控制

系统已经在各 Agent 返回值中记录 agent_events，并在 collection artifact 中记录检索 query、branch、task、cost 等信息。报告写作阶段也会记录模型、上下文长度、prompt family、fallback 是否启用等信息。

观测对象当前记录对甲方的价值
Agent 输入输出agent_events.input/output可解释每一步为什么这样做。
证据审查evidence_reviews.findings可看到哪些来源被拒绝，为什么拒绝。
覆盖审查coverage_assessments可看到维度是否充分覆盖，下一步是否补采。
claim 支持度claim_support_reviews可看到哪些结论证据不足，是否触发验证。
报告导出published_artifacts可定位 Markdown、HTML、PDF、DOCX 输出文件。

需要说明：目前 prompt 全量内容、token 明细、每个检索器耗时、重试轨迹并未完整落库，后续正式交付建议补齐。

10. 交付演示建议

面向甲方演示时，应优先展示“从证据到结论”的可信链，而不是只展示终稿。

演示步骤讲解重点推荐展示材料
输入竞品分析任务说明系统如何识别竞品、行业和分析维度。行业方向预览、active schema。
查看采集分支说明每个竞品 x 维度都有独立 branch 和 task。research_branches、research_tasks。
查看证据审查说明系统如何拒绝无 URL、错竞品、错维度或来源不足的结果。evidence_reviews。
查看补采和验证说明 coverage gap 或 weak claim 会触发回流，而不是只做末端润色。coverage_assessments、verification_task_queue。
查看最终报告说明每个重要判断应能定位 evidence ID，附录有 URL。HTML 报告、信息索引表。
